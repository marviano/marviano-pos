# Receipt and Receiptize System - Complete Guide

## Overview

The POS system implements a sophisticated multi-printer architecture with dual receipt systems: standard customer receipts and audit receiptize receipts. This document explains how the entire system works, from payment confirmation to receipt printing and numbering.

---

## System Architecture

### Multi-Printer Setup
- **Printer 1**: Receipt Printer (`receiptPrinter`) - Standard customer receipts
- **Printer 2**: Receiptize Printer (`receiptizePrinter`) - Audit/verification receipts  
- **Printer 3**: Label Printer (`labelPrinter`) - Order preparation labels

## Numbering Systems

### 1. Transaction ID (19-digit UUID)

**File**: `dist/electron/printerManagement.js` (Lines 40-79)

**Format**: `[Business(3)][YYMMDD(6)][HHMMSS(6)][Seq(4)]`

**Example**: `014241121143052001`
- `014` = Business ID (padded to 3 digits)
- `241121` = Date (24/11/21)
- `143052` = Time (14:30:52)
- `0001` = Sequence number for that second

**Purpose**: 
- Primary transaction identifier
- Shows as "Nomor Pesanan" on receipts
- Ensures uniqueness across all transactions

### 2. Database Receipt Numbers

**Database Field**: `transactions.receipt_number`

- Generated by online database when transaction is saved
- Used for daily receipt numbering
- Resets each day
- Can be null if offline-only transaction

### 3. Printer Daily Counters

**File**: `dist/electron/printerManagement.js` (Lines 84-112)

Each printer type maintains its own daily counter:

#### Counter Types:
- `receiptPrinter`: Counter for Printer 1 (1, 2, 3, 4...)
- `receiptizePrinter`: Counter for Printer 2 (1, 2, 3, 4...)
- `globalPrinter`: Overall transaction counter (1, 2, 3, 4...)

#### Counter Behavior:
- Stored in `printer_daily_counters` table
- Resets to 0 each day
- Increments only when that specific printer prints
- Independent counters for each printer type

## Printer 2: Receiptize Printer

### Purpose
- Audit and verification receipts
- Manual or automatic printing
- Used for quality control and record keeping

### Operating Modes

#### Manual Mode (Default)
- User manually selects which transactions to print
- Accessed through transaction list interface
- No automatic printing occurs

#### Auto Mode
- Automatically prints 3 out of every 10 transactions
- Random selection within each 10-transaction cycle
- Selections stored in `printer2_automation` table

### Counter Management
**File**: `src/components/PaymentModal.tsx` (Lines 1007-1013)

```typescript

if (window.electronAPI?.getPrinterCounter) {
  const counterResult = await window.electronAPI.getPrinterCounter('receiptizePrinter', 14, true);
  if (isCounterResponse(counterResult) && typeof counterResult.counter === 'number') {
    printer2Counter = counterResult.counter;
  }
}
```

### Audit Logging
**File**: `dist/electron/printerManagement.js` (Lines 200-210)

Every receiptize print is logged with:
- `transaction_id`: Which transaction was printed
- `printer2_receipt_number`: The receiptize counter number
- `print_mode`: 'auto' or 'manual'
- `cycle_number`: Which 10-transaction cycle (auto mode only)
- `global_counter`: Overall transaction counter
- `printed_at`: Timestamp of printing

### Receipt Number Resolution
**Lines 637-642**:
```typescript
const resolveReceiptSequence = (tx: Transaction): number => {
  const receiptizeCounter = receiptizeCounters[txId];
  if (typeof receiptizeCounter === 'number' && receiptizeCounter > 0) {
    return receiptizeCounter;
  }
  return typeof tx.receipt_number === 'number' ? tx.receipt_number : 0;
};
```

**Priority Order**:
1. Receiptize counter (if transaction was printed to Printer 2)
2. Database receipt_number (if available)
3. 0 (fallback)

## Sync and Offline Handling

### Dual Database Strategy
1. **Online Save**: Attempts to save to cloud database first
2. **Offline Save**: Always saves to local SQLite database
3. **Background Sync**: Queues failed online saves for retry

## Troubleshooting Common Issues

### Counter Resets
- Counters reset daily automatically
- Manual reset available through admin functions
- Each printer type has independent counters

### Missing Receipt Numbers
- Database receipt_number may be null for offline transactions
- System falls back to printer counters
- Sync resolves missing numbers when online

### Print Failures
- Failed prints queue for retry
- Audit logs track failed attempts
- Manual reprint available through transaction list

### Receiptize Not Showing Transactions
- Default view shows only receiptize-printed transactions
- Use "Show All" toggle to see complete list
- Check receiptize mode settings (auto vs manual)
